# ILIZWI Agent Guide

Last updated: 2026-03-12
Audience: feature implementers, UI builders, and review agents

This file translates the brand direction into direct instructions for build sessions.

## What the UI should feel like

Build a scholarly archival interface.

It should feel:

- serious
- calm
- high-trust
- text-led
- materially grounded

It should not feel:

- startup-hyped
- playful
- bubbly
- default dashboard
- generic SaaS admin

## The most important visual rule

The app lives between:

- a dark archival shell
- a light reading desk

The left sidebar remains dark even when the workspace is light.

## How to design screens

### Landing / shell screens

Lean into:

- Ink-Black background
- Bone-White text
- restrained accent use
- quiet, deliberate motion

### Workspace screens

Use:

- Slate-Paper as the page background
- a white reading sheet or focused work pane
- dark text optimized for long-form reading
- dark sidebar anchor

### Record detail / reader screens

Prioritize:

- source visibility
- side-by-side comparison
- metadata clarity
- annotation legibility
- calm reading margins

## Typography behavior

- use serif for major headings and archival moments
- use sans for the working interface
- let typography do the emotional work instead of decorative graphics

## Component behavior

- controls should feel precise
- cards should be quiet
- borders should be subtle
- metadata labels should be muted
- status badges should be useful, not loud

## Motion behavior

- transitions should be slow enough to feel deliberate
- hover states should ease in and out
- highlights should feel like a mark being made, not a neon flash

## What feature agents must avoid

- purple-heavy palettes
- bright blue default UI
- rounded “pill” controls everywhere
- high-contrast pure black/white combinations that vibrate visually
- flat, sterile admin panels with no reading hierarchy
- overly animated or snappy interactions

## How to handle unfinished brand work

If a feature requires styling work before final implementation polish:

- use the ILIZWI tokens already defined
- keep the structure tokenized
- do not improvise a parallel visual language

## Screen-specific reminders

### Upload flow

- should feel structured and trustworthy
- metadata entry must look rigorous, not casual

### Source viewer

- provenance must remain visible
- source and derived text should feel clearly separated

### Side-by-side reader

- must privilege reading comfort over clever layout tricks
- line length and spacing matter

### Annotation UI

- should feel like scholarly markup, not comment bubbles from a social app

### Search

- should feel library-like and precise
- filters should be clean and highly legible
