# Plan: F004 — Upload Metadata Enforcement and File Naming

**Date:** 2026-03-12
**Branch:** `codex/f004-upload-metadata-enforcement`
**Status:** PASSED — merged to main (PR #4, squash commit on 2026-03-12)

---

## Objective

Enforce structured metadata at upload intake and add a deterministic canonical naming convention (`SOURCE-PUB-YYYY-MM-DD[-vVOL][-iISS][-pPAGE][-aARTICLE]`). The canonical ref is stored in the DB alongside the UUID primary key, providing a human-readable identifier for each record.

---

## Addendums Applied (from review)

1. **Collision handling explicitly defined** — `canonical_ref` has a UNIQUE INDEX. If a collision occurs on insert (409 from Postgres), the server action appends `-r2`, `-r3`, … `-r9` until unique. If still colliding after 9 retries, return an error asking the user to disambiguate with additional fields. Collisions are expected to be rare (require same source/pub/date/volume/issue/page/article) but must be handled deterministically.

2. **Single source of truth for canonical-ref logic** — `src/lib/records/canonical-ref.ts` is a pure TypeScript module with no `"use server"` or `"use client"` directive. Both the server action (records.ts) and the client form (UploadForm.tsx) import from this shared module. The live preview in the form is generated by the same `generateCanonicalRef()` function stored to the DB — no drift possible.

3. **Fallback publication alias normalization** — Defined precisely as:
   - Strip diacritics (NFD normalize, remove combining marks via regex `\p{M}`)
   - Remove all non-alphanumeric, non-space characters
   - Remove common stop words: `la`, `le`, `les`, `the`, `a`, `an`, `of`, `de`, `du`, `der`, `die`, `das`, `van`, `y`
   - Collect remaining words, take first non-empty word
   - Truncate to 4 characters, uppercase
   - If result is fewer than 2 characters, use first 4 chars of the original title (post-diacritic strip, post-non-alnum remove), uppercase
   - If still empty (completely non-ASCII input yields nothing), use `PUB` as final fallback

4. **`canonical_ref` as NOT NULL with DEFAULT for existing rows** — The migration adds the column as `NOT NULL DEFAULT 'LEGACY'`. This backfills existing F003 test records with `LEGACY` as a sentinel value, making the invariant enforceable at DB level immediately. App-level code generates proper refs for all new uploads. The `LEGACY` sentinel is intentionally ugly and clearly distinguishable from real refs.

5. **Storage naming clarified** — F004 changes only the DB `canonical_ref` field, not the storage path. Storage path remains `{projectId}/{recordId}/{safeFilename}` (unchanged from F003). Aligning storage paths to canonical refs is deferred to a future feature (F0xx — Storage Reorganisation), noted in decisions.

## Approach

### 1. DB Migration
New file: `supabase/migrations/20260312000003_f004_canonical_ref.sql`

Add to `source_records`:
- `volume text` — nullable
- `issue_number text` — nullable
- `article_label text` — nullable
- `canonical_ref text NOT NULL DEFAULT 'LEGACY'` — backfills existing rows with sentinel

Add `UNIQUE INDEX` on `canonical_ref`.

No nullability change to `date_issued` in the DB — enforcement stays at app level (see addendum 4 note).

### 2. Canonical Ref Helper
New file: `src/lib/records/canonical-ref.ts`

Function: `generateCanonicalRef(fields): string`

**Source alias map:**
- `manual_readex` → `READEX`
- `ibali` → `IBALI`
- `nlsa` → `NLSA`
- `wits` → `WITS`

**Publication alias map (known publications):**
- `Imvo Zabantsundu` → `IMV`
- `Isigidimi samaXhosa` → `ISIG`
- `Leselinyana la Basotho` → `LB`
- Fallback: deterministic normalization (see addendum 3)

**Composition:**
`{SOURCE}-{PUB}-{YYYY-MM-DD}` then optionally append:
- `-v{vol}` if volume provided
- `-i{iss}` if issue_number provided
- `-p{page}` if page_label provided
- `-a{article}` if article_label provided

**Collision suffix (server-side only):** append `-r{n}` for n=2..9 when retrying after unique constraint violation.

### 3. Server Action — `src/lib/actions/records.ts`
- Extract new fields: `volume`, `issue_number`, `article_label`
- Add `date_issued` to required validation (return error if empty)
- Call `generateCanonicalRef()` to generate base ref
- Retry with `-r{n}` suffix on unique constraint violation (up to 9 retries)
- Insert all new fields + `canonical_ref` into `source_records`

### 4. Upload Form — `src/components/records/UploadForm.tsx`
- Make `date_issued` required (add `required` attribute + server validation)
- Add three new optional fields: Volume, Issue Number, Article Label
- Import `generateCanonicalRef` from `@/lib/records/canonical-ref`
- Live canonical ref preview: compute via `generateCanonicalRef()` as user types (useState + useEffect), shown below the metadata fields in a monospace style

### 5. Records List Page — `src/app/(app)/projects/[id]/records/page.tsx`
- Add `canonical_ref` to the select query
- Add a Canonical Ref column to the table (shown alongside Title/Archive)
- Display in a monospace font to signal it's a structured identifier

### 6. Types — `src/types/index.ts`
- Add `volume`, `issue_number`, `article_label` to `SourceRecord` as `string | null`
- Add `canonical_ref` as `string` (non-null after F004)

## Acceptance Criteria (maps to PRD test steps)

1. Upload attempt without `date_issued` blocked — error shown in UI
2. Upload attempt without any of `source_archive` / `publication_title` / `language` / `date_issued` blocked
3. Upload with full valid metadata succeeds
4. `canonical_ref` on the stored record matches expected format (e.g. `READEX-IMV-1888-01-15-p003`)
5. Records list shows both original source details (`publication_title`, `source_archive`) and `canonical_ref`
6. Two uploads with identical metadata generate distinct canonical refs (collision suffix appended)
